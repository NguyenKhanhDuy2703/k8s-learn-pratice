Khung học và thứ tự thực hành

Mục tiêu: giúp bạn hiểu rõ tại sao mỗi bước tồn tại trước khi đi vào thao tác cụ thể.

Slide này được thiết kế theo đúng trình tự một pipeline CI/CD thật: từ việc hiểu lệnh cài đặt, test, build, push, cho đến việc deploy bằng kubectl, rồi chuyển sang GitOps bằng ArgoCD.

1. Chuẩn bị trước khi bắt đầu

  • Máy local có node, npm, docker, kubectl.
  • Một cluster Kubernetes (ví dụ kind, Minikube hoặc cluster cloud).
  • Tài khoản container registry như Docker Hub hoặc GHCR để lưu image.
  • Repo GitHub để chạy GitHub Actions và lưu secret nếu cần deploy tự động.

2. Luồng học theo thứ tự

  1. Bước 1: chạy pipeline thủ công để thấy CI/CD là gì.
  2. Bước 2: chuyển thành workflow GitHub Actions để tự động hóa phần build/test/push.
  3. Bước 3: hiểu deploy bằng YAML và kubectl apply.
  4. Bước 4: nối CI + CD bằng cách “push” thay đổi vào cluster.
  5. Bước 5: tách source repo và config repo, chuẩn bị cho GitOps.
  6. Bước 6: dùng ArgoCD để “pull” trạng thái từ Git và tự đồng bộ vào cluster.
Ghi nhớ: CI là “xây dựng và kiểm chứng”, CD là “đưa ứng dụng đến môi trường”, còn GitOps là “để Git trở thành nguồn sự thật duy nhất”.
1

CI thủ công bằng tay trên terminal

Mục tiêu: hiểu pipeline thực chất chỉ là một chuỗi lệnh shell, không có gì huyền bí.

1. Tạo app Node.js đơn giản

terminal
# tạo project
mkdir hello-cicd && cd hello-cicd
npm init -y
npm install express
npm install --save-dev jest supertest
app.js
const express = require('express');
const app = express();

app.get('/', (req, res) => {
  res.send('Hello CI/CD');
});

app.get('/health', (req, res) => {
  res.json({ status: 'ok' });
});

module.exports = app;
server.js
const app = require('./app');
app.listen(3000, () => console.log('Listening on 3000'));
app.test.js
const request = require('supertest');
const app = require('./app');

test('GET / returns hello', async () => {
  const res = await request(app).get('/');
  expect(res.statusCode).toBe(200);
});

test('GET /health returns ok', async () => {
  const res = await request(app).get('/health');
  expect(res.body.status).toBe('ok');
});

2. Chạy "pipeline" bằng tay — từng lệnh một

terminal
# Bước build: cài dependencies
npm install

# Bước test: chạy unit test
npx jest

# Bước build image
docker build -t myuser/hello-cicd:v1 .

# Bước push lên registry (Docker Hub)
docker login
docker push myuser/hello-cicd:v1
Dockerfile
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm install --production
COPY . .
EXPOSE 3000
CMD ["node", "server.js"]
Quan sát: mỗi lần bạn sửa code, bạn phải tự lặp lại 4 lệnh trên theo đúng thứ tự — install → test → build → push. Nếu quên chạy test, bug có thể bị push thẳng vào image. Đây chính là vấn đề CI giải quyết.
Checklist trước khi qua bước 2
Rút ra
Pipeline = chuỗi lệnh có thứ tự cố định (install → test → build → push). CI tool không làm gì "ma thuật" — nó chỉ chạy đúng các lệnh này tự động mỗi khi có code mới.
2

Tự động hóa CI với GitHub Actions

Mục tiêu: biến 4 lệnh thủ công ở bước 1 thành workflow tự động mỗi khi push code.

1. Push project lên GitHub

terminal
git init
git add .
git commit -m "initial commit"
git remote add origin https://github.com/<user>/hello-cicd.git
git push -u origin main

2. Tạo workflow CI

.github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]

jobs:
  build-test-push:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Install dependencies
        run: npm install

      - name: Run tests
        run: npx jest

      - name: Log in to Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKERHUB_USERNAME }}
          password: ${{ secrets.DOCKERHUB_TOKEN }}

      - name: Build and push image
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: |
            myuser/hello-cicd:latest
            myuser/hello-cicd:${{ github.sha }}
Lưu ý: thêm 2 secret DOCKERHUB_USERNAME và DOCKERHUB_TOKEN trong Settings → Secrets and variables → Actions trên GitHub repo.

3. Trigger và quan sát

  • Sửa nhẹ app.js (đổi text trả về), commit và push.
  • Vào tab Actions trên GitHub, theo dõi từng step chạy — chính là các lệnh bạn đã chạy tay ở bước 1.
  • Sau khi xong, kiểm tra Docker Hub: image mới với tag github.sha đã xuất hiện.
Checklist trước khi qua bước 3
Rút ra
GitHub Actions chỉ là "máy chạy lại các lệnh terminal của bạn" mỗi khi có sự kiện (push). CI = automation của bước build/test/push. Phần "deploy" vẫn chưa có gì cả.
3

CD thủ công bằng kubectl

Mục tiêu: hiểu "deploy" thực ra là thay đổi desired state trong cluster, không phải "chạy chương trình".

1. Tạo cluster local

terminal
# dùng kind (Kubernetes in Docker)
kind create cluster --name dev

# kiểm tra
kubectl get nodes

2. Viết manifest Deployment + Service

k8s/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: hello-cicd
spec:
  replicas: 2
  selector:
    matchLabels:
      app: hello-cicd
  template:
    metadata:
      labels:
        app: hello-cicd
    spec:
      containers:
        - name: hello-cicd
          image: myuser/hello-cicd:v1
          ports:
            - containerPort: 3000
k8s/service.yaml
apiVersion: v1
kind: Service
metadata:
  name: hello-cicd
spec:
  selector:
    app: hello-cicd
  ports:
    - port: 80
      targetPort: 3000
  type: NodePort

3. Apply thủ công và xem thực tế

terminal
kubectl apply -f k8s/deployment.yaml
kubectl apply -f k8s/service.yaml

# xem pod đang chạy
kubectl get pods
kubectl get deployment hello-cicd

4. "Deploy phiên bản mới" = sửa tag rồi apply lại

terminal
# build & push image v2 (lặp lại bước 1/2 với tag mới)
docker build -t myuser/hello-cicd:v2 .
docker push myuser/hello-cicd:v2

# sửa tay file deployment.yaml: image: myuser/hello-cicd:v2
# rồi apply lại
kubectl apply -f k8s/deployment.yaml

# quan sát rolling update diễn ra
kubectl rollout status deployment/hello-cicd
kubectl get pods -w
Quan sát quan trọng: bạn không "restart app" — bạn chỉ thay đổi 1 dòng YAML rồi apply. Kubernetes tự so sánh state hiện tại với state mong muốn trong YAML và tự điều chỉnh (rolling update pod cũ → pod mới).
Checklist trước khi qua bước 4
Rút ra
"Deploy" = khai báo state mong muốn (YAML) rồi gửi cho API server. Việc còn lại — chạy container, rolling update — là Kubernetes tự làm (đúng như đã học ở kiến trúc controller/kubelet).
4

CD tự động kiểu "push-based"

Mục tiêu: ghép CI (bước 2) + CD (bước 3) lại — pipeline tự deploy sau khi build xong.

1. Chuẩn bị kubeconfig cho GitHub Actions

PowerShell
# PowerShell (Windows): tạo chuỗi base64 từ kubeconfig
[Convert]::ToBase64String([System.IO.File]::ReadAllBytes("$HOME\.kube\config"))

Lưu kết quả vào secret KUBE_CONFIG_B64 trong GitHub repo. Nếu bạn dùng Git Bash / WSL, lệnh tương đương là cat ~/.kube/config | base64 -w 0.

2. Thêm step deploy vào workflow

.github/workflows/ci.yml (thêm vào cuối job trước)
      # ... các step build-test-push như bước 2 ...

      - name: Set up kubeconfig
        run: |
          mkdir -p ~/.kube
          echo "${{ secrets.KUBE_CONFIG_B64 }}" | base64 -d > ~/.kube/config

      - name: Update image tag and deploy
        run: |
          sed -i "s|image: myuser/hello-cicd:.*|image: myuser/hello-cicd:${{ github.sha }}|" k8s/deployment.yaml
          kubectl apply -f k8s/deployment.yaml
          kubectl apply -f k8s/service.yaml
          kubectl rollout status deployment/hello-cicd

3. Trigger toàn bộ flow end-to-end

  • Sửa app.js lần nữa → commit → push.
  • Workflow: test → build → push image → tự deploy lên cluster.
  • Theo dõi kubectl get pods -w trên máy local để thấy pod mới xuất hiện ngay sau khi Action chạy xong.
Hạn chế của mô hình này: GitHub Actions cần có quyền truy cập trực tiếp vào cluster (kubeconfig). Với cluster production thật, việc lưu credential trong CI là rủi ro bảo mật lớn — đây là lý do GitOps (bước 5-6) ra đời.
Checklist trước khi qua bước 5
Rút ra
Đây là CD truyền thống — "push-based": CI tool chủ động gửi lệnh vào cluster. Hoạt động tốt, nhưng cluster phải mở quyền cho hệ thống bên ngoài → vấn đề bảo mật & khó audit "ai/khi nào đã thay đổi gì".
5

Tách repo code và repo manifest (GitOps)

Mục tiêu: làm cho Git trở thành nguồn sự thật duy nhất — chuẩn bị cho ArgoCD.

1. Tạo repo thứ 2 — "config repo"

terminal
# repo mới, chỉ chứa manifest
mkdir hello-cicd-config && cd hello-cicd-config
git init
mkdir k8s
# copy deployment.yaml + service.yaml từ bước 3 vào đây
git add .
git commit -m "initial manifests"
git remote add origin https://github.com/<user>/hello-cicd-config.git
git push -u origin main

Giờ bạn có 2 repo riêng biệt:

  • hello-cicd — chứa source code app (repo bước 1-2)
  • hello-cicd-config — chỉ chứa manifest YAML (single source of truth cho cluster)

2. Sửa workflow: không deploy trực tiếp, chỉ cập nhật config repo

.github/workflows/ci.yml (thay step deploy ở bước 4)
      # ... build-test-push như cũ ...

      - name: Checkout config repo
        uses: actions/checkout@v4
        with:
          repository: <user>/hello-cicd-config
          token: ${{ secrets.CONFIG_REPO_TOKEN }}
          path: config-repo

      - name: Avoid self-trigger loop
        run: |
          # Nếu workflow nằm cùng repo với manifest, hãy thêm paths-ignore để tránh commit mới kích hoạt lại workflow
          # Ví dụ trong .github/workflows/ci.yml:
          # on:
          #   push:
          #     branches: [main]
          #     paths-ignore:
          #       - 'k8s/**'

      - name: Update image tag in manifest
        run: |
          cd config-repo
          sed -i "s|image: myuser/hello-cicd:.*|image: myuser/hello-cicd:${{ github.sha }}|" k8s/deployment.yaml
          git config user.name "ci-bot"
          git config user.email "ci-bot@example.com"
          git add k8s/deployment.yaml
          git commit -m "chore: bump image to ${{ github.sha }}"
          git push
Lưu ý: CONFIG_REPO_TOKEN là Personal Access Token có quyền push vào repo hello-cicd-config — tạo trong GitHub Settings → Developer settings.

3. Trigger và quan sát

  • Push code vào hello-cicd → CI build/test/push image như cũ.
  • Bước cuối không còn gọi kubectl — thay vào đó tạo 1 commit mới trong hello-cicd-config với tag image mới.
  • Mở repo hello-cicd-config, xem commit log: mỗi lần deploy = 1 commit, có timestamp, có message rõ ràng.
Quan trọng: tại bước này, cluster chưa tự cập nhật! Commit trong config repo chỉ thay đổi Git, chưa ai apply nó vào cluster. Đây chính là "khoảng trống" mà ArgoCD sẽ lấp vào ở bước 6.
Checklist trước khi qua bước 6
Rút ra
Git repo hello-cicd-config giờ là "desired state" tập trung — bất kỳ ai nhìn vào repo này đều biết chính xác cluster "nên" chạy phiên bản nào. Đây là nền tảng của GitOps.
6

ArgoCD — pull-based CD

Mục tiêu: để ArgoCD tự động làm bước "apply" mà ở bước 5 vẫn còn thiếu.

1. Cài ArgoCD vào cluster

terminal
kubectl create namespace argocd
kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml

# truy cập UI qua port-forward
kubectl port-forward svc/argocd-server -n argocd 8080:443

# lấy password admin ban đầu
$secret = kubectl -n argocd get secret argocd-initial-admin-secret -o jsonpath="{.data.password}"; 
[System.Text.Encoding]::UTF8.GetString([System.Convert]::FromBase64String($secret))

2. Tạo Application — trỏ về config repo ở bước 5

GitOps tip: với mô hình App of Apps, có thể tạo một root-app.yaml duy nhất để trỏ vào thư mục k8s/ hoặc apps/. Nếu bạn dùng ArgoCD để quản lý các app con, hãy luôn thêm finalizers: - resources-finalizer.argocd.argoproj.io để khi xóa root app, ArgoCD tự dọn tài nguyên con theo chuỗi (cascading delete), tránh rác resource.
argocd-app.yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: hello-cicd
  namespace: argocd
  finalizers:
    - resources-finalizer.argocd.argoproj.io
spec:
  project: default
  source:
    repoURL: https://github.com/<user>/hello-cicd-config.git
    targetRevision: main
    path: k8s
  destination:
    server: https://kubernetes.default.svc
    namespace: default
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
terminal
kubectl apply -f argocd-app.yaml
Quan trọng: nếu workflow của bạn nằm trong cùng repo chứa manifest, hãy thêm paths-ignore: ['k8s/**'] trong .github/workflows/ci.yml để tránh trigger vòng lặp vô tận: commit cập nhật manifest → workflow chạy lại → lại commit mới → lặp mãi.

3. Lặp lại flow end-to-end và quan sát sự khác biệt

  • Push code mới vào hello-cicd → CI build/test/push image → commit tag mới vào hello-cicd-config (giống bước 5).
  • Không cần làm gì thêm — trong vòng ~3 phút (hoặc bấm "Refresh" trên UI), ArgoCD tự phát hiện commit mới và tự kubectl apply vào cluster.
  • Mở ArgoCD UI, xem trạng thái Synced / OutOfSync và lịch sử sync.
selfHeal: true — nếu ai đó vô tình kubectl edit trực tiếp vào cluster (sửa khác với Git), ArgoCD sẽ tự động revert lại đúng theo trạng thái trong Git. Thử nghiệm: chạy kubectl scale deployment hello-cicd --replicas=5 rồi xem ArgoCD tự đưa về lại số replicas trong YAML.

So sánh: Bước 4 (push) vs Bước 6 (pull / ArgoCD)

Đây chính là khác biệt cốt lõi giữa CD truyền thống và GitOps.

Tiêu chí Bước 4 — Push-based Bước 6 — ArgoCD (pull-based)
Ai khởi tạo deploy CI tool (GitHub Actions) chủ động gọi kubectl apply ArgoCD trong cluster tự kéo (pull) thay đổi từ Git
Credential cluster Phải lưu kubeconfig trong CI secrets Không cần — cluster không expose credential ra ngoài
Nguồn sự thật (source of truth) Trạng thái CI pipeline + cluster (có thể lệch nhau) Git repo — luôn khớp 100% nhờ selfHeal
Audit / rollback Xem log CI pipeline Git history — rollback = git revert
Thay đổi tay (drift) Không phát hiện được Tự phát hiện và tự sửa (selfHeal)

Sau khi đi qua đủ 6 bước, ArgoCD không còn là "hộp đen" — nó chỉ tự động hóa đúng phần việc bạn đã làm tay (commit ở bước 5 → kubectl apply ở bước 3), nhưng theo hướng pull thay vì push.

Tóm tắt cuối buổi
  • CI giúp bạn kiểm chứng và đóng gói code an toàn trước khi deploy.
  • CD giúp bạn đưa phiên bản mới vào Kubernetes theo một quy trình có thể lặp lại.
  • GitOps giúp bạn giữ cluster luôn đúng với Git, dễ audit, dễ rollback và dễ tự động hóa.

7

Control Plane — Bộ não của cluster

Câu hỏi cốt lõi: Ai quyết định pod chạy ở đâu, và quyết định đó được lưu ở đâu?

1. Xem các thành phần Control Plane đang chạy

Control Plane cũng chạy dưới dạng pod — trong namespace kube-system. Nhìn vào đó để thấy rõ từng thành phần.

PowerShell
# Xem toàn bộ pod hệ thống
kubectl get pods -n kube-system

# Lọc riêng từng thành phần
kubectl get pods -n kube-system | Select-String -Pattern 'etcd'
kubectl get pods -n kube-system | Select-String -Pattern 'kube-apiserver'
kubectl get pods -n kube-system | Select-String -Pattern 'kube-scheduler'
kubectl get pods -n kube-system | Select-String -Pattern 'kube-controller'
Ghi chú cho cụm lớn: kube-scheduler có thể chỉ xét một phần số node để giảm tải, theo tham số percentageOfNodesToScore (thường khoảng 50% trong cụm lớn). Điều này giúp tối ưu tốc độ lập lịch trên cluster nhiều node.
Quan sát: Tên pod có dạng kube-apiserver-<tên-node>. Điều đó nói lên điều gì? API Server gắn với node nào? Nếu node đó chết thì sao?
Kiến thức nâng cao: khi một Pod được tạo, kube-scheduler không "chạy Pod" ngay lập tức. Nó đi qua 3 pha: Filtering (loại node không đủ tài nguyên/không thỏa taints/affinity), Scoring (đánh giá ưu tiên như image locality, topology spread), rồi Binding (ghi nodeName vào etcd). Dòng Successfully assigned... chỉ là thông báo Pod đã được phân node, chưa phải lúc container bắt đầu chạy.

2. Theo dõi API Server nhận request thế nào

Bật verbose để thấy mọi HTTP call mà kubectl thực sự gửi đi. Đây là cách nhìn thấu vào "bên dưới" kubectl.

PowerShell
# PowerShell: bật verbose tracing để quan sát HTTP request/response
kubectl get pods --v=8 2>&1 | Select-String -Pattern 'GET|POST|Response'

# Xem stream watch event
kubectl get pods --watch --v=6 2>&1 | Select-Object -First 20

3. Xem etcd — nơi lưu toàn bộ trạng thái cluster

PowerShell
# PowerShell: lấy tên pod etcd
$ETCD_POD = kubectl get pod -n kube-system -l component=etcd -o jsonpath='{.items[0].metadata.name}'

# Dùng etcdctl đọc key từ etcd
kubectl exec -n kube-system $ETCD_POD -- sh -c "ETCDCTL_API=3 etcdctl --endpoints=https://127.0.0.1:2379 --cacert=/etc/kubernetes/pki/etcd/ca.crt --cert=/etc/kubernetes/pki/etcd/server.crt --key=/etc/kubernetes/pki/etcd/server.key get /registry --prefix --keys-only" | Select-Object -First 30

# Nếu dùng Minikube / image distroless và lỗi 'sh not found', chạy trực tiếp etcdctl thay vì sh -c
kubectl exec -n kube-system $ETCD_POD -- etcdctl --endpoints=https://127.0.0.1:2379 --cacert=/var/lib/minikube/certs/etcd/ca.crt --cert=/var/lib/minikube/certs/etcd/server.crt --key=/var/lib/minikube/certs/etcd/server.key get /registry --prefix --keys-only | Select-Object -First 30
Thử nghiệm tư duy: Nếu xóa một key trong etcd, điều gì sẽ xảy ra với pod tương ứng? Đừng làm thật — chỉ cần nghĩ qua để hiểu etcd là "nguồn sự thật duy nhất" của cluster.

4. Quan sát kube-scheduler ra quyết định

PowerShell
# Xem log của scheduler
kubectl logs -n kube-system -l component=kube-scheduler --tail=50

# Tạo pod pending bằng tài nguyên quá lớn
@'
apiVersion: v1
kind: Pod
metadata:
  name: pod-unpschedulable
spec:
  containers:
  - name: app
    image: nginx
    resources:
      requests:
        memory: "999Gi"
'@ | kubectl apply -f -

# Xem event của pod
kubectl describe pod pod-unpschedulable | Select-String -Pattern 'Events|Warning|FailedScheduling|Unschedulable'

# Dọn dẹp
kubectl delete pod pod-unpschedulable
Takeaway — Bước 7
  • Control Plane gồm 4 thành phần: API Server (cổng vào duy nhất), etcd (database), Scheduler (chọn node), Controller Manager (đảm bảo trạng thái đúng với mong muốn).
  • Mọi thứ trong cluster — pod, service, configmap — đều chỉ là một entry trong etcd.
  • kubectl thực chất chỉ là HTTP client gọi vào API Server.
8

Worker Node & Pod — Bên trong một container đang chạy

Câu hỏi cốt lõi: kubelet làm gì sau khi nhận pod spec? Container Runtime thực sự là gì?

1. Xem thông tin node và tài nguyên

terminal
# Xem toàn bộ node trong cluster
kubectl get nodes -o wide

# Xem chi tiết một node: OS, kernel, container runtime, CPU/RAM
kubectl describe node <tên-node>

# Xem tài nguyên đang được dùng trên từng node
kubectl top nodes

2. Hiểu kubelet — agent chạy trên mỗi node

kubelet là process chạy trực tiếp trên node (không phải pod). Nó nhận pod spec từ API Server và ra lệnh cho Container Runtime tạo container.

PowerShell
# Trên node (nếu dùng kind hoặc minikube SSH vào được)
docker exec -it kind-control-plane sh   # kind
# hoặc
minikube ssh                              # minikube

# Bên trong node — xem kubelet process
Get-Process kubelet
systemctl status kubelet

# Xem kubelet đang watch những gì từ API Server
journalctl -u kubelet --since "5 minutes ago" | Select-String -Pattern 'pod|sync' | Select-Object -Last 20
Thực tế trên Linux: khi dùng Get-Process kubelet hoặc ps aux | grep kubelet, bạn có thể thấy cả các tiến trình khác như kube-apiserver, node_exporter, prometheus-operator. Đây là hiện tượng bình thường vì các tiến trình giám sát hoặc dịch vụ bảo mật có chứa từ khóa liên quan đến kubelet.

3. Tạo pod và theo dõi từng giai đoạn lifecycle

PowerShell
# Terminal 1: watch pod status realtime
kubectl get pod -w

# Terminal 2: tạo pod bằng Here-String (PowerShell)
@"
apiVersion: v1
kind: Pod
metadata:
  name: arch-demo
spec:
  initContainers:
  - name: init-wait
    image: busybox
    command: ['sh', '-c', 'echo Init done; sleep 2']
  containers:
  - name: main
    image: nginx
    ports:
    - containerPort: 80
    readinessProbe:
      httpGet:
        path: /
        port: 80
      initialDelaySeconds: 3
"@ | kubectl apply -f -

# Quan sát: Pending → Init → PodInitializing → Running

# Xem events chi tiết — thấy rõ từng bước kubelet làm
kubectl describe pod arch-demo | Select-String -Pattern 'Events|Normal|Warning|Failed|Started'
Events cho thấy: Scheduled (scheduler chọn node) → Pulling (kubelet kéo image) → Pulled → Created → Started. Mỗi dòng là một hành động thực tế của kubelet.
Init container & readiness: initContainers phải chạy xong hoàn toàn trước khi container chính bắt đầu; readinessProbe quyết định Pod có được phép nhận traffic hay không. Nếu bạn dùng bộ gõ tiếng Việt và nhập lệnh trực tiếp vào shell, có thể bị lỗi mã hóa UTF-8; chuyển sang tiếng Anh trước khi gõ là tốt nhất.

4. Exec vào container — xem namespace isolation

PowerShell
# Vào bên trong container đang chạy
kubectl exec -it arch-demo -- sh

# Bên trong container:
hostname                    # tên pod
cat /etc/hosts              # thấy IP của pod
ps aux                      # chỉ thấy process của container này
cat /proc/1/net/dev         # network interface riêng của pod
exit

5. Hiểu tại sao pod có IP riêng

PowerShell
# Xem IP của từng pod
kubectl get pods -o wide

# Debug nhanh từ container tạm
kubectl run test-curl --image=curlimages/curl --rm -it --restart=Never -- curl http://<IP-của-arch-demo>:80

# Dọn dẹp
kubectl delete pod arch-demo
Takeaway — Bước 8
  • kubelet nhận pod spec → gọi Container Runtime Interface (CRI) → tạo container.
  • Pod là một nhóm container dùng chung network namespace — nên chúng thấy nhau qua localhost.
  • Events trong kubectl describe là nhật ký thực tế của kubelet, rất hữu ích để debug.
  • Trong image slim/alpine, lệnh ps hoặc sh có thể không có sẵn; nếu cần debug mạng, hãy đọc /proc/1/net/dev trực tiếp.

Networking — Service, CoreDNS và định tuyến nội bộ

Mục tiêu: hiểu vì sao Pod cần Service, tại sao CoreDNS là “danh bạ nội bộ” của cluster, và NodePort giúp mở dịch vụ ra ngoài thế nào.

1. Vì sao phải dùng Service?

Pod là ephemeral: nó có thể chết và tạo mới bất kỳ lúc nào, nên địa chỉ IP của Pod không ổn định. Nếu app khác hardcode IP Pod thì sẽ mất kết nối sau mỗi restart hoặc rolling update.

Service giống “quầy lễ tân” với một IP ổn định: khách hàng gọi vào Service, Service sẽ chuyển traffic đến các Pod đang chạy ở phía sau theo selector/label.
PowerShell
# Expose Deployment bằng Service
kubectl expose deployment web --port=80 --target-port=80

# Mặc định là ClusterIP nếu không ghi --type
kubectl get svc web -o wide

# --port = cổng của Service, --target-port = cổng container

2. CoreDNS: DNS nội bộ của cluster

CoreDNS đóng vai trò như cuốn danh bạ DNS nội bộ. Khi một Service được tạo, CoreDNS cập nhật bản ghi ánh xạ tên Service sang ClusterIP. Do đó, các Pod chỉ cần gọi tên Service thay vì hardcode IP Pod.

PowerShell
# Xem CoreDNS đang chạy
kubectl get pods -n kube-system | Select-String coredns

# Test DNS từ trong cluster
kubectl run dns-test --image=busybox:1.36 --rm -it --restart=Never -- nslookup web

# Tên đầy đủ: service.namespace.svc.cluster.local
kubectl run dns-test --image=busybox:1.36 --rm -it --restart=Never -- nslookup web.default.svc.cluster.local
Lợi ích: CoreDNS lưu IP của Service, không phải IP của từng Pod. Khi scale out nhiều Pod, hệ thống vẫn ổn định vì chỉ có những bản ghi Service cần duy trì.

3. NodePort — mở cổng ra ngoài

NodePort cho phép mở cổng tĩnh trên Worker Node để hệ thống bên ngoài truy cập dịch vụ. Tuy nhiên, nó làm việc ở tầng 4 (TCP/UDP), nên không có khả năng hiểu HTTP path/host như Ingress.

PowerShell
# Tạo Service kiểu NodePort
kubectl expose deployment web --name=web-nodeport --port=80 --target-port=80 --type=NodePort

# Xem cổng được cấp
kubectl get svc web-nodeport

# Nếu dùng Minikube
minikube service web-nodeport
Ghi nhớ: NodePort dùng cổng trong dải 30000-32767. Trong production, thường dùng LoadBalancer hoặc Ingress để có routing và bảo mật tốt hơn.

4. Gỡ lỗi Windows PowerShell

  • Kubernetes phân biệt hoa thường nghiêm ngặt: viết sai NodePort thành NodePor sẽ bị lỗi Unsupported value.
  • Trên PowerShell, lệnh Bash có dấu \ để xuống dòng có thể bị lỗi parser. Nên nối thành một dòng hoặc dùng backtick ` khi cần xuống dòng.
  • Để mở Service qua NodePort trên Windows, dùng minikube service <tên-service> để tự mở trong trình duyệt.
Takeaway — Mạng Kubernetes
  • Service là IP ổn định và load balancer nội bộ cho các Pod thay đổi liên tục.
  • CoreDNS là DNS nội bộ của cluster, giúp microservices gọi nhau bằng tên Service thay vì hardcode IP Pod.
  • NodePort mở cổng ra ngoài Node ở tầng 4, phù hợp dev/testing hoặc tiền đề cho production.
9

Networking — Service, kube-proxy và CoreDNS (phần mở rộng)

Câu hỏi cốt lõi: Tại sao không dùng IP của pod trực tiếp? Service giải quyết vấn đề gì?

1. Vấn đề của IP pod — thực hành để thấy

terminal
# Tạo Deployment 2 replicas
kubectl create deployment web --image=nginx --replicas=2
kubectl get pods -o wide        # ghi nhớ 2 IP pod

# Xóa một pod → nó tự tạo lại với IP MỚI
kubectl delete pod <tên-pod-1>
kubectl get pods -o wide        # IP đã thay đổi!
Vấn đề thực tế: Pod có thể chết và tái sinh bất kỳ lúc nào với IP khác. Nếu app khác hardcode IP của pod, nó sẽ mất kết nối. Service giải quyết điều này.
ClusterIP mặc định: nếu bạn chạy kubectl expose deployment web --port=80 --target-port=80 mà không truyền --type, Kubernetes sẽ tạo Service loại ClusterIP. Đây là IP nội bộ, chỉ các Pod trong cluster mới gọi được. --port là cổng của Service; --target-port là cổng thực của container.
Mô hình mạng nội bộ: Pod-to-Pod trong cùng Node có thể giao tiếp trực tiếp qua IP gốc (flat network), không cần NAT. Cấu trúc bên dưới là veth pair + virtual bridge (ví dụ cni0 hoặc docker0) để chuyển gói tin giữa các Pod.

2. Tạo Service và hiểu cơ chế selector

Selector / Label: Service không quan tâm đến tên Pod; nó tìm Pod bằng label. Nếu Pod có label khớp với selector, nó sẽ được đưa vào Endpoints của Service. Đây là cơ chế giúp Service luôn trỏ đúng nhóm backend đang chạy.
terminal
# Expose Deployment bằng Service ClusterIP
kubectl expose deployment web --port=80 --target-port=80

# Xem Service — chú ý CLUSTER-IP
kubectl get service web

# Xem endpoints — Service đang trỏ tới những pod nào?
kubectl get endpoints web

# Xóa một pod → đợi pod mới lên → check endpoints lại
kubectl delete pod <tên-pod>
kubectl get endpoints web       # Endpoints tự cập nhật!

3. DNS nội bộ — coreDNS trong cluster

CoreDNS: là add-on nội bộ của cluster, giúp các Pod tìm Service bằng tên (web.default.svc.cluster.local) thay vì phải hardcode IP. CoreDNS lắng nghe thay đổi từ API Server và cập nhật bản ghi DNS theo thời gian thực.
PowerShell
# Xem coreDNS đang chạy
kubectl get pods -n kube-system | Select-String coredns

# Resolve tên Service bằng busybox
kubectl run dns-test --image=busybox:1.36 --rm -it --restart=Never -- nslookup web

# Resolve tên đầy đủ
kubectl run dns-test --image=busybox:1.36 --rm -it --restart=Never -- nslookup web.default.svc.cluster.local

4. Các loại Service — ClusterIP vs NodePort vs LoadBalancer

Phân biệt proxy: kubectl proxy là công cụ trên máy của bạn để truy cập API Server; kube-proxy là daemon chạy trên mỗi Node, dùng để định tuyến gói tin giữa Service và Pod. Hai thứ này khác nhau hoàn toàn về vai trò.
terminal
# NodePort — expose ra ngoài cluster qua port trên node
kubectl expose deployment web --name=web-nodeport \
  --port=80 --target-port=80 --type=NodePort

kubectl get service web-nodeport
# EXTERNAL-IP = <none>, nhưng PORT(S) = 80:3xxxx/TCP
# Truy cập: http://<IP-của-node>:3xxxx

# Dùng minikube service để mở thẳng trên browser
minikube service web-nodeport  # nếu dùng minikube

# Xem kube-proxy tạo iptables rule thế nào (trên node)
iptables -t nat -L KUBE-SERVICES | Select-String -Pattern 'web'
Takeaway — Bước 9
  • Service là một IP ổn định + DNS name, đứng trước một nhóm pod — không bị ảnh hưởng khi pod chết/tái sinh.
  • kube-proxy chạy trên mỗi node, dịch ClusterIP → Pod IP thực bằng iptables hoặc ipvs.
  • DNS trong cluster do coreDNS quản lý — mọi Service đều có tên dạng <name>.<namespace>.svc.cluster.local.

Storage — ConfigMap, Secret, PersistentVolume

Câu hỏi cốt lõi: Dữ liệu của app sẽ đi về đâu khi pod bị xóa?

1. ConfigMap — biến môi trường không hardcode vào image

terminal
# Tạo ConfigMap
kubectl create configmap app-config \
  --from-literal=APP_ENV=production \
  --from-literal=LOG_LEVEL=info

kubectl get configmap app-config -o yaml

# Dùng ConfigMap làm env trong pod
cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: Pod
metadata:
  name: config-demo
spec:
  containers:
  - name: app
    image: busybox
    command: ['sh', '-c', 'env | findstr APP; sleep 3600']
    envFrom:
    - configMapRef:
        name: app-config
EOF

kubectl logs config-demo   # thấy APP_ENV=production

2. Secret — lưu thông tin nhạy cảm

terminal
# Tạo Secret
kubectl create secret generic db-creds \
  --from-literal=username=admin \
  --from-literal=password=supersecret

# Xem Secret — giá trị bị base64 encode
kubectl get secret db-creds -o yaml

# Decode để thấy giá trị thật (PowerShell)
$pwd = kubectl get secret db-creds -o jsonpath='{.data.password}'
[System.Text.Encoding]::UTF8.GetString([System.Convert]::FromBase64String($pwd))
Lưu ý bảo mật: Secret trong K8s mặc định chỉ encode base64 — không encrypt. Trong môi trường production cần bật Encryption at Rest hoặc dùng Vault.

3. PersistentVolume — dữ liệu sống sót qua vòng đời pod

terminal
# Tạo PersistentVolumeClaim
cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: data-pvc
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 100Mi
EOF

kubectl get pvc   # trạng thái Bound — được gán vào PV tự động

# Mount vào pod
cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: Pod
metadata:
  name: storage-demo
spec:
  containers:
  - name: app
    image: busybox
    command: ['sh', '-c', 'echo "hello persistent" > /data/test.txt; cat /data/test.txt; sleep 3600']
    volumeMounts:
    - mountPath: /data
      name: my-vol
  volumes:
  - name: my-vol
    persistentVolumeClaim:
      claimName: data-pvc
EOF

# Xóa pod, tạo pod mới cùng PVC — dữ liệu vẫn còn
kubectl delete pod storage-demo
kubectl apply -f -  <<EOF
... (pod mới cùng PVC) ...
EOF
kubectl exec -it storage-demo -- cat /data/test.txt
Takeaway — Bước 10
  • ConfigMap: config không nhạy cảm → inject vào pod qua env hoặc volume file.
  • Secret: thông tin nhạy cảm → cần thêm biện pháp bảo mật ngoài K8s trong production.
  • PVC/PV: dữ liệu tồn tại độc lập với pod — pod chết, dữ liệu còn; pod mới mount lại là tiếp tục được.

RBAC — Ai được làm gì trong cluster

Câu hỏi cốt lõi: Làm sao giới hạn để ArgoCD chỉ deploy được vào namespace của nó, không phải toàn cluster?

1. Kiểm tra quyền hiện tại của mình

terminal
# Bạn có quyền gì?
kubectl auth can-i --list

# Kiểm tra quyền cụ thể
kubectl auth can-i create pods
kubectl auth can-i delete nodes
kubectl auth can-i get secrets -n kube-system

2. Tạo ServiceAccount bị giới hạn quyền

terminal
# Tạo ServiceAccount
kubectl create serviceaccount limited-user -n default

# Tạo Role — chỉ được get/list pods trong namespace default
cat <<EOF | kubectl apply -f -
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: default
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list", "watch"]
EOF

# Bind Role vào ServiceAccount
kubectl create rolebinding pod-reader-binding \
  --role=pod-reader \
  --serviceaccount=default:limited-user \
  -n default

3. Kiểm chứng quyền bằng impersonation

terminal
# Giả vờ mình là limited-user — xem được gì
kubectl auth can-i --list \
  --as=system:serviceaccount:default:limited-user

# Được: get pods
kubectl get pods \
  --as=system:serviceaccount:default:limited-user

# Không được: xóa pods
kubectl delete pod arch-demo \
  --as=system:serviceaccount:default:limited-user
# → Error from server (Forbidden)

# Không được: xem secret
kubectl get secrets \
  --as=system:serviceaccount:default:limited-user
# → Error from server (Forbidden)

4. Hiểu RBAC của ArgoCD — liên hệ lại bước 6

terminal
# Xem ClusterRole mà ArgoCD dùng
kubectl get clusterrole argocd-application-controller -o yaml | Select-String -Pattern 'rules:|apiGroups|resources|verbs'

# Xem tất cả RBAC của ArgoCD
kubectl get rolebinding,clusterrolebinding -n argocd | Select-String -Pattern 'argocd'

# ServiceAccount mà ArgoCD dùng để gọi API Server
kubectl get serviceaccount -n argocd
Kết nối với bước 6: ArgoCD không dùng kubeconfig như bạn. Nó dùng ServiceAccount với ClusterRole cho phép nó get/list/watch/apply mọi resource — đó là lý do nó có thể tự kubectl apply mà không cần credential từ bên ngoài.
Takeaway — Bước 11
  • RBAC = Role (quyền gì) + RoleBinding (gán cho ai) + Subject (User / Group / ServiceAccount).
  • Dùng kubectl auth can-i --as=... để kiểm tra quyền mà không cần đăng nhập thật.
  • Mọi pod đều chạy dưới một ServiceAccount — nếu không khai báo, dùng default SA (thường có quyền tối thiểu).

Lộ trình tiếp theo — từ Architecture đến Production

Sau khi đã hands-on qua 5 chủ đề kiến trúc, đây là những gì bạn nên khám phá tiếp theo để đến mức "hiểu cluster thật sự".

Chủ đề Tại sao quan trọng Hands-on gợi ý
Ingress & Ingress Controller Layer 7 routing — một IP vào nhiều service khác nhau theo path/hostname Cài nginx-ingress, tạo Ingress rule, quan sát nginx config tự sinh
HPA & VPA Auto-scaling pod dựa trên CPU/memory/custom metric Dùng kubectl autoscale, chạy load test bằng hey, xem pod tự nhân lên
Network Policy Firewall giữa các pod — mặc định pod nào cũng nói chuyện được với nhau Deny all ingress → từng bước allow từng service → kiểm chứng bằng curl
Helm Package manager cho K8s — thay vì apply từng YAML Cài một chart có sẵn, rồi tự viết chart đơn giản với values.yaml
Multi-node cluster thật kind/minikube che mất nhiều thứ — cần thấy node thật, taint/toleration, node affinity Dùng k3s trên 2-3 VPS rẻ (~$5/node), cài thủ công từng bước
Thứ tự học được đề xuất

  2. Dựng cluster k3s 2 node thật — làm lại toàn bộ từ bước 7 trên môi trường thật.
  3. Thêm Ingress + HPA vào app từ bước 6 (ArgoCD deploy + tự scale).
  4. Học Network Policy để hiểu tại sao microservices cần isolation.
  5. Học Helm để đóng gói ứng dụng chuẩn hơn thay vì raw YAML.
